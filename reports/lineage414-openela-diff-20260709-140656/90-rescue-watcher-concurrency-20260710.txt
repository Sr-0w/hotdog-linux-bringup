# 90 - Rescue watcher concurrency hardening

Date: 2026-07-10 04:10 CEST

Scope:

Offline hardening only. No phone command, flash, reboot, ADB, fastboot, or SSH
cycle was performed for this change.

Problem:

The phone is currently invisible to the host, so the active rescue path matters.
The existing `phone-operation.lock` protects the actual flash/reboot window, but
the launcher and supervisor could still be started twice from different labels
or shells. That made the orchestration noisier than needed and could leave a
watcher stopped after a transient lock collision.

Changes:

```text
scripts/rescue-boot-b-when-visible.sh
  - moves phone-lock acquisition to the main decision points
  - retries without touching the phone when the lock is busy
  - acquires the lock before recovery ADB artifact collection/reboot handoff

scripts/start-stable-rescue-watcher.sh
  - adds a serial-level start lock under logs/manual-rescue-watchers/
  - includes the serial in pid/log filenames
  - refuses a duplicate rescue watcher for the same serial by default
  - adds --allow-duplicate for intentional diagnostics

scripts/watch-rescue-visible-supervisor.sh
  - adds a per-serial/label instance lock
  - refuses a duplicate supervisor for the same serial/label by default
  - keeps its passive behavior: no flash, no reboot, no phone-operation lock
```

Current active processes were left running. The hardening applies to future
starts and to any restart launched by the current supervisor.

Validation target:

```text
bash -n scripts/rescue-boot-b-when-visible.sh \
  scripts/start-stable-rescue-watcher.sh \
  scripts/watch-rescue-visible-supervisor.sh
```
