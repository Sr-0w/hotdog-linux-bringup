# 92 - USB rescue guard hardening

Date: 2026-07-10 04:32 CEST

Scope:

Offline orchestration hardening plus a host-side watcher that can be left armed
while the phone is invisible. No phone-visible action was performed during the
patch/validation step because the host currently sees no ADB, fastboot, USB
network, or Qualcomm device.

Problem:

`rescue-boot-b-when-visible.sh` already watches fastboot and recovery ADB. The
separate USB watcher can also notice fastboot, Qualcomm 9008, Qualcomm 900e, or
other Android/Qualcomm USB identities, but its fastboot restore path delegated
to `restore-pmos-boot-b-from-fastboot.sh`, which did not take the shared
`phone-operation.lock`.

That made the helper less suitable to run beside the existing rescue watcher:
two watchers could notice fastboot in the same poll window and both try to own
the restore path.

Changes:

```text
scripts/restore-pmos-boot-b-from-fastboot.sh
  - sources phone-lock.sh
  - adds --lock-wait SEC, default 0
  - takes phone-operation.lock before checking/flashing/rebooting fastboot
  - exits 75 when the lock is busy, leaving the target untouched
  - releases the lock through an EXIT trap

scripts/rescue-boot-b-when-usb-visible.sh
  - adds --label and --allow-duplicate
  - adds a per-serial/label instance lock under logs/manual-rescue-watchers/
  - refuses duplicate USB rescue watcher instances by default
  - avoids process-substitution / command-substitution self-detection false positives
  - matches --serial/--label with fixed token-boundary checks instead of broad globs
  - writes current.run/current.log symlinks only from the instance that owns the watcher lock
  - treats fastboot restore exit 75 as a transient busy-lock condition and keeps watching
  - keeps EDL write disabled unless --edl-write is explicit

scripts/current-autopilot-status.sh
  - reports any running rescue-boot-b-when-usb-visible.sh process
  - prints the active USB rescue watcher run log through current.run when available, not a newer duplicate-probe log

scripts/watch-rescue-visible-supervisor.sh
  - avoids the same command-substitution self-detection false positive in future starts
  - uses the same fixed token-boundary matching for --serial/--label duplicate checks
```

Validation:

```text
find scripts -type f -name '*.sh' -print0 | xargs -0 -n1 bash -n
  OK

./scripts/validate-current-candidates.sh
  all current candidates validated

./scripts/restore-pmos-boot-b-from-fastboot.sh --serial b6bd2252 --after-restore none --timeout 1 --lock-wait 0
  took and released phone-operation.lock
  refused to act because b6bd2252 is not visible in fastboot

./scripts/rescue-boot-b-when-usb-visible.sh --serial b6bd2252 --timeout 1 --poll 1 --label validation-probe --no-edl-write
  state=none
  timed out without touching the phone

start-stop-daemon ... rescue-boot-b-when-usb-visible.sh --label host-usb-guard --no-edl-write
  running PID 2272770
  state=none
  phone-operation.lock absent

./scripts/watch-rescue-visible-supervisor.sh ... --label stable-guard
  Supervisor already running for b6bd2252/stable-guard: PID 2080483
```

Operational note:

This watcher is a complement, not a replacement. `rescue-boot-b-when-visible.sh`
remains the primary fastboot/recovery ADB rescue path. The USB watcher adds
visibility for USB states outside ADB/fastboot and can safely coexist because
fastboot restore now shares the same phone-operation lock.
