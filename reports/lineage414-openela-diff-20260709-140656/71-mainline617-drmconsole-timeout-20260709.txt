## Interpretation

The mainline kernel is accepted by the boot stack well enough to avoid immediate
fastboot rejection, but the boot does not reach a recoverable USB path or a
visible DRM-console path.

The DRM-console helper can only render if the kernel reaches initramfs and
creates `/dev/dri/card0`. This test did not prove either condition. The most
useful reading is therefore:

```text
likely blocker: before initramfs helper starts, or before mainline can create DRM card0
not explained by: missing userspace shell alone
not fixed by: injecting the validated downstream DRM console into the ramdisk
```

## Follow-up

- Do not retest this exact `220500` image without a new kernel or DTB
  hypothesis.
- Keep downstream `215005` as the current boot_b baseline.
- For mainline, prioritize kernel-entry evidence, DTB/boot ABI differences, or
  ramoops/pstore reliability before adding more userspace payloads.
- If visible output is attempted again, add a non-DRM fallback only if the
  kernel is known to expose a framebuffer-like device early enough.
