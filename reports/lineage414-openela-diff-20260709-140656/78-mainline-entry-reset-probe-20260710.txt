mainline entry reset probe - 2026-07-10
=======================================

Reason
------

The mainline 6.17 tests that avoid immediate fastboot return still produce no
visible output, USB gadget, initramfs watchdog marker, or pstore. That points
below initramfs-side instrumentation.

The smallest next mainline probe is to make `primary_entry` request an
immediate PSCI reset. This tests whether the bootloader branches into the
kernel at all.

Prepared patch
--------------

```text
/home/srobin/dev/hotdog/patches/experimental-mainline-entry-psci-reset-probe.patch
```

Local validation:

```text
git -C /home/srobin/dev/hotdog/src/kernel/linux-postmarketos-qcom-sm8150-v6.17.0-sm8150 apply --unidiff-zero --check /home/srobin/dev/hotdog/patches/experimental-mainline-entry-psci-reset-probe.patch
```

Probe logic:

```asm
movz x0, #0x0009
movk x0, #0x8400, lsl #16
mov  x1, xzr
mov  x2, xzr
mov  x3, xzr
smc  #0
wfe loop if reset returns
```

Evidence for `smc`:

```text
stock entry12 DTB pack: compatible = "arm,psci-1.0"; method = "smc";
mainline sm8150.dtsi: method = "smc";
PSCI_0_2_FN_SYSTEM_RESET = 0x84000009
```

Expected interpretation
-----------------------

```text
quick reset or quick fastboot/recovery return:
  the bootloader entered the kernel and reached primary_entry

same no-USB/no-fastboot hang:
  likely before primary_entry, in bootloader handoff, or in an exception around
  the SMC path itself

normal Linux boot:
  the patch was not actually present in the tested kernel
```

Do not apply this patch to the normal mainline worktree permanently. Use it
only for a dedicated disposable probe build and keep the downstream `215005`
restore path armed. A RAM marker probe is less disruptive if a recovery-side
memory dump path is available; this reset probe is kept as a binary fallback.
