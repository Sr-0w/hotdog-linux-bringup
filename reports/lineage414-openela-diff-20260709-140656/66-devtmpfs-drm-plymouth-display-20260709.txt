Good:
- /dev/dri/card0 and /dev/dri/renderD128 are now created automatically.
- drmdevice identifies the platform DRM node as /soc/qcom,mdss_mdp@ae00000,

Still not solved:
- /dev/fb0 is still absent.
- simple-framebuffer still fails early:
  simple-framebuffer chosen:framebuffer@9c000000: No memory resource
- modetest exits immediately after setting a mode, so it only flashes the
  backlight briefly before the connector returns to disabled.
- kmscube starts EGL/GLES through llvmpipe but exits quickly and does not keep
  DSI-1 enabled.

- plymouthd remained alive as root.

Reboot validation:
- A first reboot showed that starting Plymouth directly from local.d was too
  early; the daemon exited and DSI-1 returned to disabled.
- The hook was changed to start a background waiter that waits for /dev/dri/card0
  and DSI-1 connected before starting Plymouth.

Next implications
-----------------
For visual boot:
- Use Plymouth/DRM as the practical display path for this downstream kernel.
- Do not rely on /dev/fb0 unless the downstream msm_drm fbdev path is patched
  or a different framebuffer handoff is fixed.

For mainline:
- The proven requirement is no longer merely "make USB work"; the userspace
  display path needs a real DRM node and a compositor/splash to hold the panel.
- The simplefb reserved-memory problem remains worth investigating, but it is
  not required for a visible pmOS userspace splash once DRM is available.
