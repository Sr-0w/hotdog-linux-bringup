mainline entry RAM marker probe - 2026-07-10
============================================

Reason
------

The mainline 6.17 failures are still below the normal initramfs/debug
instrumentation layer: no useful USB, no pstore, no watchdog marker, and no
visible userspace output. A reset probe can tell whether `primary_entry` is
reached, but it is destructive and only binary.

This probe writes stage markers directly into the existing ramoops reserved
window. If recovery or another boot path can dump `/dev/mem`, the raw bytes can
show how far the arm64 entry path progressed.

Marker window
-------------

```text
physical base: 0xa9800000
window size:   0x400000
collector:     dd if=/dev/mem bs=4096 skip=694272 count=1024
```

Existing evidence:

```text
arch/arm64/boot/dts/qcom/sm8150-oneplus-hotdog.dts:
  ramoops@a9800000 reg = <0 0xa9800000 0 0x400000>

scripts/build-watchdog-bootimg.sh:
  ramoops.mem_address=0xa9800000 ramoops.mem_size=0x400000

scripts/collect-recovery-crash-artifacts.sh:
  adb_exec_out_capture ramoops-phys-a9800000-4m.img 'dd if=/dev/mem bs=4096 skip=694272 count=1024 2>/dev/null'
```

Markers
-------

```text
offset 0x0: ENT1 = 0x454e5431, after preserve_boot_args
offset 0x4: ENT2 = 0x454e5432, after __cpu_setup, before __primary_switch
offset 0x8: SWT3 = 0x53575433, at __primary_switch entry, before __enable_mmu
```

Each marker is written as a 32-bit word, then the first 0x40 bytes of the
ramoops window are cleaned to PoC through `dcache_clean_poc`.

Expected interpretation
-----------------------

```text
ENT1 only:
  bootloader entered primary_entry and preserve_boot_args returned, but the
  entry path failed before or inside CPU setup / idmap work.

ENT1 + ENT2:
  CPU setup returned, but failure happened around the branch to
  __primary_switch or at the start of MMU enable.

ENT1 + ENT2 + SWT3:
  __primary_switch was reached; the next suspect is __enable_mmu or later
  early mapping/relocation.

no markers:
  the bootloader likely did not branch to this kernel entry, the image was not
  the tested payload, the ramoops window was cleared before collection, or the
  early store/cache-clean path faulted before persistence.
```

Collection note
---------------

After a failed boot that returns to recovery with ADB, run the existing crash
collector and search the dumped `ramoops-phys-a9800000-4m.img` for the marker
bytes. In a raw byte view, the 32-bit words appear little-endian:

```text
ENT1: 31 54 4e 45
ENT2: 32 54 4e 45
SWT3: 33 54 57 53
```

This patch is not for normal boot images. It is a disposable diagnostic for a
dedicated mainline probe build.
