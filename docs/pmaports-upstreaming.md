# pmaports upstreaming plan

## Scope and evidence baseline

The initial upstream target is `device/testing` with a close-to-mainline SM8150
kernel. Submission is gated on direct boot of the exact image generated by the
proposed pmaports packages, following the [direct-boot procedure](direct-boot.md).

The historical r0 evidence proves that the K1 kernel package path can build.
The intermediate r3 double-build evidence confirms identical transformed DTB,
modules, and `modules.builtin`, but not byte-identical APK or `vmlinuz` output:
the latter differs by 29 bytes from the GNU build ID and three initramfs CPIO
mtimes, while `.PKGINFO` build dates vary. The current r4 fixes
`SOURCE_DATE_EPOCH=1761609785`; two builds in the tested pmbootstrap environment
produced byte-identical `27,172,035`-byte APKs with SHA256
`74d7cff718be9a06b8858360fe56c1ccd8d1fd7653151546b0480029694d803e`.
This does not prove cross-toolchain reproducibility, hardware equivalence, a
clean standard pmaports submission build, or direct boot of the
package-generated image. See the
[project status](status.md), [historical r0 package evidence](evidence/k1-kernel-package.md),
and [K1 DTB evidence](evidence/k1-dtb-source.md).

Relevant pmaports policy references are:

- *Device Categorization*, sections "Testing" and "Community";
- *Packaging Guidelines*, sections "Maintainership" and "Package versioning";
- *Firmware Packages*, sections "User choice" and "Packaging";
- *Kernel Command Line Configuration*, section "Migration from deviceinfo";

## Confirmed blockers on 2026-07-12

The current package set is a bring-up workspace, not a submission-ready pmaports
branch. The confirmed blockers are:

- rebase the proposed packages cleanly onto `upstream/main` at `2d43d15ac` or
  its reviewed successor, then rebuild against that tree's current
  `devicepkg-dev` contract;
- install mkinitfs integration below `/usr/share/mkinitfs`, not legacy paths
  below `/etc`, and remove package behavior implemented through `/etc/local.d`;
- extend the shared `linux-postmarketos-qcom-sm8150` package with the maintained
  hotdog DTS and required source fixes instead of submitting dependencies on
  the forensic K1 package or downstream bridge;
- make required firmware dependencies explicit, resolve ownership conflicts
  with existing firmware packages, and document provenance and redistribution
  terms for every shipped file;
- exclude the permissive bring-up `doas` policy and the unvalidated WirePlumber
  policy from the submission; both remain laboratory-only inputs.

## Prerequisites for `device/testing`

- [ ] The normal device package flow produces a boot image that direct-boots on
  hardware without a laboratory repack step.
- [ ] The device reaches a writable root filesystem with working UFS, mainline
  modules, USB NCM or ACM access, SSH, and clean reboot to bootloader and
  recovery.
- [ ] The port and every dependency build from a clean checkout without forced,
  lax, or skipped validation. This is required by *Device Categorization*,
  section "Testing".
- [ ] The shared kernel satisfies the close-to-mainline, supported-version, and
  LLVM requirements in *Device Categorization*, section "Testing".
- [ ] Every APKBUILD uses a valid `maintainer="Name <email>"` assignment and a
  valid SPDX license. A `# Maintainer:` comment is not a substitute for the
  `maintainer` variable.
- [ ] New packages start at `pkgrel=0`; later device-package revisions follow
  *Packaging Guidelines*, section "Package versioning".
- [ ] The device package and `deviceinfo` pass current pmaports policy checks.
  Boot-image fields must match the validated header-v2 contract, DTB handling,
  page size, offsets, partition size, `super` target, and A/B behavior.
- [ ] Kernel arguments are supplied through `kernel-cmdline.conf`, as required
  by *Kernel Command Line Configuration*, rather than the deprecated
  `deviceinfo_kernel_cmdline` field. Debug arguments must be justified or
  removed.
- [ ] DRM capability is declared only after the validated DRM/DSI/panel path is
  functional. Other optional capabilities follow the same evidence rule.
- [ ] A public `oneplus-hotdog` wiki page is verified or created with accurate
  installation, recovery, and known-issues information before submission.

The migrated version-2 `deviceinfo` and `kernel-cmdline.conf` containing
`clk_ignore_unused` pass `dint` structurally. That result validates metadata
shape only. `deviceinfo_drm` must remain absent from the submission candidate
until DRM, DSI, panel, and userspace scanout work on hardware.

## Expected package architecture

### `device-oneplus-hotdog`

Place the device package in `device/testing`. It should contain only device
metadata, `deviceinfo`, `kernel-cmdline.conf`, direct dependencies on the shared
kernel and firmware package, and narrowly justified runtime integration.

Any required daemon must use a normal OpenRC service. Package hooks that create
`/etc/local.d` scripts, blanket passwordless privilege rules, or development
watchers are not acceptable substitutes for a maintained service or a kernel
fix.

### `firmware-oneplus-hotdog`

The device package should depend directly on this firmware package. Package only
firmware not already supplied by `linux-firmware` or another pmaports package,
with reviewable provenance, redistribution terms, source URLs, and checksums.
Check file ownership and conflicts against existing firmware packages.

The `20241212-r0` split has completed usrmerge packaging validation: its
eight APKs contain 16 payloads, all under `/usr/lib/firmware`. This is not
runtime peripheral validation and does not resolve the provenance,
redistribution, or ownership-conflict blockers above.

Do not create a `nonfree-firmware` subpackage. This follows *Firmware Packages*,
sections "User choice" and "Packaging". Partition loading may be used only where
the documented pmaports mechanism and the validated device layout support it.

### `linux-postmarketos-qcom-sm8150`

Extend the existing shared SM8150 kernel package instead of submitting the K1
forensic kernel as a second device-specific mainline package. The extension
should contain the maintained DTS, required driver patches, and minimal config
fragments. Kernel configuration should follow *Fragment-Based Kernel
Configuration* and remain suitable for other SM8150 devices.

The binary DTB transform sequence documented in the
[K1 DTB evidence](evidence/k1-dtb-source.md) is investigative evidence, not the
final implementation. Changes still required after direct-boot testing must be
expressed as reviewable DTS or kernel source changes and validated with the
shared package.

The experimental ARM64 entry-layout change remains conditional. If direct-boot
comparison proves it necessary, its scope and upstream suitability require
separate justification; otherwise it must be omitted.

Recommended review order:

1. Extend and validate `linux-postmarketos-qcom-sm8150`.
2. Add `firmware-oneplus-hotdog` with provenance and conflict checks.
3. Add `device-oneplus-hotdog` and generate the final installation artifacts.

## Excluded laboratory and bridge artifacts

The initial mainline submission must exclude:

- the standalone `linux-oneplus-hotdog-mainline617-k1` forensic package;
- the downstream 4.14 bridge and `linux-oneplus-hotdog-lineage414` dependency;
- stale 6.4 and 6.8.7 kernel snapshots;
- generated boot images, binary DTB packs, partition dumps, extracted private
  device data, and full diagnostic logs;
- external DTB mutation scripts or binary-only fixes;
- custom launchers, watchers, bring-up privilege rules, and permissive `doas`
  configuration;
- unvalidated WirePlumber, audio, desktop, or convenience policy changes.

Downstream 4.14 remains diagnostic and recovery evidence only, as recorded in
the [status matrix](status.md). If a downstream port is ever proposed, it must be
a separate `device/downstream` submission with `-downstream` naming and must not
be presented as the mainline target.

Repository inclusion rules remain those in the
[repository layout and artifact policy](repository-layout.md).

## CI and validation sequence

1. Run `apkbuild-lint` on every changed APKBUILD and `dint check` on the final
   `deviceinfo`. Record `dint` as structural validation only; it cannot prove
   a declared hardware capability.
2. Verify source checksums, SPDX identifiers, firmware redistribution terms,
   package ownership, and dependency closure.
3. Run `shellcheck` on every submitted shell script or OpenRC service helper.
4. Build `firmware-oneplus-hotdog`, `linux-postmarketos-qcom-sm8150`, and
   `device-oneplus-hotdog` with standard `pmbootstrap build` commands from a
   clean source state. Do not use force, lax, or skip options as evidence.
5. Run the normal `pmbootstrap kconfig check` for the shared kernel. Enable the
   package's config checks as described by *kconfigcheck*.
6. Build DTBs with schema validation and run the applicable `dtbs_check` set.
   Treat new warnings introduced by the hotdog DTS as failures.
7. Generate installation and boot images through the normal device-package
   flow. Inspect the boot header, DTB placement, offsets, image size, partition
   target, module layout, and AVB assumptions against the validated contract.
8. Direct-boot that exact package output and record the criteria from
   [direct-boot](direct-boot.md): writable root, UFS, USB NCM or ACM, SSH,
   mainline modules, and clean reboot paths.
9. Repeat the build and hardware smoke test from a clean state. Publish only
   reproducible hashes and redacted evidence permitted by the
   [artifact policy](repository-layout.md).
10. Verify all local documentation links and the public wiki entry before the
    merge request is opened.

Hardware operations must follow the [device safety rules](device-safety.md).

## Promotion to `community`

Promotion is a later milestone, not part of the initial testing submission. It
requires all criteria in *Device Categorization*, section "Community", including:

- maintained installation documentation on the postmarketOS wiki;
- a close-to-mainline kernel that passes `pmbootstrap kconfig check --community`;
- automatic kernel upgrade support and, for an Android boot-image workflow,
  validated `deviceinfo_flash_kernel_on_update=true` behavior;
- successful release-upgrade handling and maintenance across the required
  kernel-release history;
- a kernel version within the policy's allowed age;
- an active maintainer with regular access to the device.

The project promotion gate additionally requires the staged validation in the
[hardware roadmap](hardware-roadmap.md): full RAM, Apps SMMU, UFS ICE,
DRM/DSI/panel, touch, then Wi-Fi and USB host. Any remaining limitation must be
measured, documented in the [status matrix](status.md), and explicitly acceptable
under the current pmaports community policy before promotion is requested.
