# Evidence: 2026-07-11 K1 mainline cycle

This page is a public, versioned summary derived from local ignored logs,
build manifests, and image manifests from the 2026-07-11 K1 cycle. It is a
text evidence record, not a redistributable boot artifact.

Local absolute paths, device serial numbers, host interface names, usernames,
and raw capture contents are intentionally omitted. No password or private key
material is included.

## Result

- The exact K1 Linux 6.17 payload reached postmarketOS userspace through the
  downstream 4.14 kexec bridge.
- USB NCM, USB ACM detection, and SSH were observed under mainline.
- A K1-matching `qcom-wdt.ko` module loaded after userspace and registered the
  Qualcomm watchdog device.
- A mainline `RESTART2(bootloader)` request caused a physical reboot, but the
  device returned to the normal persistent bridge boot instead of entering
  fastboot. This is consistent with missing boot-mode mapping.
- Temporary `fastboot boot` did not establish a direct mainline boot path.
  The next direct-boot milestone is a persistent `boot_b` test.

## ACM no-echo capture

The corrected ACM collector is passive: it does not use adb, fastboot, SSH,
USB reset, or payload writes to the tty device. Its tty reader opens the ACM
node read-only with `O_NOCTTY` and nonblocking mode, applies raw termios, and
clears host-side echo before reading.

For the K1 kexec run, the collector matched `ttyACM0`, captured 1049 bytes,
and produced no reader stderr. A later D1 direct-boot collector was armed with
the same `18d1` / `0617` criteria and did not observe a matching `ttyACM`
device because the temporary `fastboot boot` run did not leave the original
fastboot USB instance.

## qcom-wdt module and reboot

The watchdog module artifact was built for the exact K1 ABI:

- module: `qcom-wdt.ko`
- SHA256: `23e7abab0f150b969e031ba0030f2270ec2900c97d316028073330e746faab36`
- vermagic: `6.17.0-sm8150-g379d8fe35c7c-dirty SMP preempt mod_unload aarch64`
- `CONFIG_QCOM_WDT=m`
- `CONFIG_MODVERSIONS` disabled
- `CONFIG_MODULE_SIG` disabled

The live run copied the same module hash to the device, verified the matching
vermagic, inserted the module after mainline userspace was reachable, and then
observed:

- `qcom_wdt` listed as a live module
- `/sys/class/watchdog/watchdog0` pointing at `17c10000.watchdog`
- `/dev/watchdog` and `/dev/watchdog0` present but not opened by the test
- `qcom_wdt` probe of `17c10000.watchdog` returning `0`

At 16:43:49, a `RESTART2(bootloader)` request was issued from the mainline
boot. Fastboot did not become visible within the 30-second observation window.
The device later returned over postmarketOS SSH on the downstream 4.14 bridge,
which validates a physical reboot but not bootloader-mode selection. A
subsequent downstream 4.14 bootloader request did enter fastboot, so the
remaining gap is specific to the mainline restart-mode handoff.
